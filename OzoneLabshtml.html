この章は管理者向けです

Ozoneのプロトコル:

Ozoneではアクセスの対象や利用方法に応じ、いくつかのプロトコルをサポートします。

オペレーション

FSO

Hive / Spark / other

ofs://

o3fs://

s3a://

Ozoneはマルチプロトコルストレージであり、以下のようなインターフェイスをサポートします。

  • ofs: Hadoop互換ファイルシステム。HDFSライクなインターフェイスを提供し、アプリケーションを変更することなくOzoneにアクセスできます。Spark、YARN、Hiveといったエンジンは変更なくOzone上で動作します。
  • s3a: Amazon S3(Simple Storage Service)プロトコル。S3クライアントやS3のSDKで開発されたアプリケーションでOzoneにアクセスできます。
  • s3aを使用する場合、全ての通信がS3Gatewayを通過することに注意してください。
  • o3fs: ofsの旧バージョン(現在は推奨していません)
  • o3: Ozoneシェルコマンド向けのオブジェクトストアインターフェイス

Ozone CLI:

Ozone CLIは”ozone”コマンドとそのサブコマンドで構成されます。

  • ozone fs - FSO(File System Optimized)バケットに対して、HDFS互換コマンドを実行します。ofsおよびo3fsインターフェイスに対応しています。
  • ozone sh - Ozoneを操作するシェルコマンドインターフェイス。操作対象のオブジェクトは、Volume、Bucket、Keyです。

Ozone fs

ozone fs コマンドを実行してみましょう。

操作の概要:
  • HDFSへのアクセス
  • Ozoneへのアクセス
  • ボリュームの作成
  • バケットの作成
  • バケットへのデータ保存
  • バケット上のデータ削除
操作の詳細:

Cloudera Manager上で、OzoneにHDFS依存関係が設定されると、OzoneクライアントライブラリのサポートがHDFSクライアントコマンドに組み込まれ、Ozoneでも利用可能となります。Ozoneのプロトコルスキームを指定することで、”hdfs dfs”コマンドでもOzoneにアクセスできるようになります。

  • ozone fsコマンドでHDFSをリストします。
    パラメータを何も指定しない場合、ozone fsはHDFSへアクセスします。

$ ozone fs -ls /

  • Ozoneファイルシステムをリストします。Ozoneへアクセスするには、パスの先頭にOzone sidを付加します。Ozone sid(Ozoneサービス名)は”ozone”です。

$ ozone fs -ls ofs://ozone/

ボリュームはOzoneファイルシステムの最上位の概念であり、キーを保存するバケットを管理します。クオータやユーザ権限はボリュームに対して適用されます。

  • ボリューム”vol1”を作成します。:

$ ozone fs -mkdir ofs://ozone/vol1

  • もう一度、Ozoneファイルシステムをリストすると、ボリュームの一覧がリストされます。

$ ozone fs -ls ofs://ozone/

  • ボリューム”vol1”にバケット”bucket1” を作成します。ファイルはバケットに保存されます。

$ ozone fs -mkdir ofs://ozone/vol1/bucket1

  • バケットのファイルをリストします。

$ ozone fs -ls ofs://ozone/vol1/bucket1

OFSでは、先頭の2階層のボリュームとバケットをディレクトリのように表示し、従来のファイルシステムのスタイルを踏襲します。ただし、ボリュームの直下にキー(ファイル)を置くことはできません。キー(ファイル)を置くと、ディレクトリパス形式のキーにアップロードされます。バケットのルートに保存されたファイル名である場合もあれば、バケット内のディレクトリパスとファイル名である場合もあります。ボリュームとバケットには命名規則があり、特定の文字や大文字小文字の使用が禁止されている点に注意が必要です。キーにはこの制限はありません。また、OFSを使用する場合は必ず”/ボリューム名/バケット名”から指定する点に注意してください。ファイルシステムには最低2つのディレクトリが必要です（/tmpはHDFSとの互換性のためボリューム直下にファイルを置くことができます）。また、EC(Erasure Coding)や暗号化はバケットレベルでの設定となります。

  • バケットに任意のディレクトリを作成しましょう。

$ ozone fs -ls ofs://ozone/vol1/bucket1/{名前}

  • /var/logディレクトリのログを、バケット”bucket1”の直下に置きます

$ ozone fs -put /var/log/dnf.log ofs://ozone/vol1/bucket1/{名前}

  • /vo1/bucket1に置かれたファイルを確認します:

$ ozone fs -ls ofs://ozone/vol1/bucket1/{名前}

  • キーファイル内のコンテンツを確認します

$ ozone fs -cat ofs://ozone/vol1/bucket1/{名前}/dnf.log | head

ozone fs コマンドを使ってOzoneに保存されたファイルを削除しても、すぐには削除されません。ファイルは削除したユーザがアクセス可能なゴミ箱に移動されます。ゴミ箱は隠しフォルダであり”/user/ユーザ名/.Trash/Current”のようなディレクトリです(先頭に”.”が付くことに注意)。

  • .Trashフォルダにファイルを置かずにゴミ箱への移動をスキップする場合は、-skipTrashオプションを追加すると、ファイルは直ちに削除されます。

$ ozone fs -rm -r -skipTrash ofs://ozone/vol1/bucket1/{ユーザ名}/dnf.log

現在のCloudera Base on premiseでは、デフォルトFS(fs.defaultFS)をOzoneにすることはできません。そのため、ozone fsコマンドでファイルを相対パス指定した場合はHDFS上のユーザホームが表示されます(/user/bobなど)。

Ozone上にユーザホームに相当するディレクトリを作成する場合は、常にOzone sidとボリューム名、バケット名を指定する必要があります。

  • 引数なしで”ozone fs -ls”を実行します。

$ ozone fs -ls

drwxr-xr-x   - bob cdp-users          0 2025-08-29 07:09 testfile

  • bobユーザのホームディレクトリにあるtestfileに対してfsckを実行します。

ファイルがHDFS上の”/user/bob/testfile”であることがわかります。

$ hdfs fsck testfile | tail -1

Connecting to namenode via https://cdp.13.158.161.107.nip.io:9871/fsck?ugi=bob&path=%2Fuser%2Fbob%2Ftestfile

The filesystem under path '/user/bob/testfile' is HEALTHY

  • Ozone上のユーザホームをリストします。
    これは絶対パス指定なのでファイルがOzone上にあることが確認できます。

$ ozone fs -ls ofs://ozone/user/bob

-rw-rw-rw-   1 bob bob          0 2025-09-19 08:04 ofs://ozone/user/bob/testfile

  • -fsオプションでデフォルトのファイルシステムを指定することもできます。
    その場合は、Ozone上のファイルへ相対パスでアクセスすることができます。

$ ozone fs -fs ofs://ozone -ls

-rw-rw-rw-   1 bob bob          0 2025-09-19 08:04 testfile

Ozone sh

もう一つのシェルコマンドであるozone shコマンドを使ってみます

操作の概要:
  • ボリュームの作成
  • バケットの作成
  • ボリューム、バケットの削除
  • リスト
  • ボリューム、バケット、キーの情報取得
  • クオータの設定
  • シンボリックリンク

この他に、Erasure Coding(EC)、レプリケーション、バケットレイアウトの設定などの操作もあります

  • ボリューム、バケットを削除します。
    この操作のために、/my-volume1/my-bucket3というボリューム、バケットを予め作成しておきます

# この操作はエラーとなります。バケットが残っているとボリュームを削除できません。

$ ozone sh volume delete o3://ozone/my-volume1

# ボリュームを削除するには先にバケットを削除しておきます。

$ ozone sh bucket delete o3://ozone/my-volume1/my-bucket3

  • リスト

$ ozone sh volume list --user=admin

$ ozone sh bucket setquota --namespace-quota=10 --space-quota 100MB o3://ozone/my-volume1/my-bucket1

# クオータの削除

$ ozone sh volume clrquota --namespace-quota o3://ozone/my-volume1

$ ozone sh bucket clrquota --space-quota o3://ozone/my-volume1/my-bucket1

  • シンボリックリンク

# これはバケットへのシンボリックリンクです。

# バケットを作成せずにボリューム直下にリンクを作成します。

$ ozone sh bucket link o3://ozone/my-volume1/my-bucket1 o3://ozone/my-volume1/my-bucket3

ラボ3 セキュリティ

Rangerの有効化と設定

  • Ozoneでオブジェクトの権限管理を行うにはRangerとの統合が必要です。
    Cloudera ManagerのOzoneサービスロール画面に移動し、”RANGER Service”がチェックされていることを確認します。
    (ワークショップ環境はすでにチェック済みです。設定だけ確認します。)

  • Ranger WebUIに管理者ユーザでログインします。
  • Ozoneサービスの下の”cm_ozone”を選択します。
  • リストの最初にある”all - volume, bucket and key”のペンアイコンをクリックします。
  • 許可設定”Allow Condition”の最初の設定行に自分のユーザを追加します。
  • 画面の最下部までスクロールしてsaveボタンを押して保存します。

  • 複雑な権限を付与してみます。以下のように権限を付与します。
  • ボリューム、バケットのread権限(createとdeleteはなし)
  • Ozone Volume: ボリューム名または”*”
  • bucket: バケット名または”*”
  • key: none
  • Allow Conditions: ユーザを指定し、read, listをチェック
  • キーに対する、deleteも含む全ての権限の付与
  • Ozone Volume: ボリューム名または”*”
  • bucket: バケット名または”*”
  • key: none
  • Deny Conditions: ユーザを指定し、create, write, deleteをチェック
  • このような複雑な権限設定を行う場合は、テストユーザを作成し実際にアクセスして確認するといいでしょう。下図はbobユーザの設定例です。


  • 管理者などを想定し、Ozoneの全領域へのアクセス件を付与する場合は、以下のようなポーリシーを作成します。
  • Ozone Volume:”*”
  • bucket:”*”
  • key: “*”
  • Allow Conditions: ユーザを指定し、全ての権限をチェック

ラボ4 高度な機能

Ozone bucket Erasure coding  ⇒ Workshop環境では利用できません

Ozoneはファイルシステム内のレプリケーション方式として、RATISとErasure Codingをサポートしています。

  • デフォルトのレプリケーションタイプはRATISです。Ozoneのストレージコンテナを常に3つ保持します。実際に保存したいデータの3倍のディスク領域が必要です。
  • Erasure Coding(EC)は、符号理論に基づく誤り訂正技術により、損失したデータブロックをパリティブロックから復元します。パリティブロックの設定により幾つかのパターンがありますが、実際に保存したいデータの概ね1.5倍前後のディスク領域が必要です。

レプリケーションタイプはバケット単位に設定し、バケット作成時に明示することでECが有効かされ、それ以外はRATISとなります。

ラボ5 FSOバケットとOBSバケット

バケットタイプとプロトコルの互換性

Ozoneは以下のような複数のバケットレイアウトをサポートします。

  • FILE_SYSTEM_OPTIMIZED (FSO):
  • 階層型ファイル システム名前空間で、ファイルとディレクトリの概念があります
  • アトミックなリネーム・削除をサポートします。
  • Hadoopファイルシステム互換インターフェイスを提供します。
  • HiveやImpalaなどHadoop APIに対応したエンジンに最適です。
  • ゴミ箱(.Trash)機能も実装されています。
  • OBJECT_STORE (OBS):
  • S3のようなフラットなkey&valueの名前空間です。
  • S3インターフェイスを使用するアプリケーション向けです。
  • LEGACY:
  • 古いバージョンのOzoneで使われていたバケットタイプです。
  • Hadoopファイルシステム互換です。

ラボ6 Hive・Sparkの利用

権限設定

RangerでのhiveユーザのOzoneへのアクセスおよびomユーザのHiveServer2へのアクセス許可が必要です。

HiveServer2の設定では、OzoneおよびHDFS双方のマネージドDBへのアクセスを設定します。

Rangerポリシーの設定

  • RangerのHadoopSQLで、全てのデータベース、テーブル、カラムに対して、全ての権限を付与します。“all - database, table, column”ポリシーで、Allow Conditionsの最初のエントリーにユーザを追加します。

  • 次にHive URLポリシーに権限付与します。”all-url”ポリシーで、Allow Conditionsの最初のエントリーにユーザを追加します。

  • “hive”ユーザとSparkの実行ユーザとなる”yarn”ユーザがOzoneの全てのオブジェクトにアクセスできるよう権限を設定します。”cm_ozone”の“all - volume, bucket, key”のAllow Conditionsの最初のエントリーに”hive”, “yarn”ユーザを追加します。

HiveServer2の設定

  • OzoneとHDFSの双方でHiveマネージド データベース&テーブルを作成する場合は、設定の追加が必要です。

ラボ7 HDFS ⇔ Ozoneのデータコピー

権限設定

  • Ranger WebUIにログインし、”HDFS”サービスの”cm_hdfs”の”all-path”ポリシーを編集します。
  • distcpを実行するユーザに(R)ead, (W)rite, e(X)ecuteの権限を付与します。
  • 画面下部までスクロールしSaveボタンで保存します。

.

操作の詳細:

  • Ozone CLIのコピーコマンドでHDFS〜Ozone間のファイルコピーを行います

# 外部サイトからファイルをダウンロードしHDFSに保存します

$ wget -qO - https://www.fueleconomy.gov/feg/epadata/vehicles.csv | hdfs dfs -copyFromLocal - /tmp/vehicles.csv

# ozone fs -cpコマンドでHDFSからOzoneへファイルをコピーします

$ ozone fs -mkdir -p ofs://ozone/vol1/bucket1/cp/vehicles

$ ozone fs -cp hdfs:///tmp/vehicles.csv ofs://ozone/vol1/bucket1/cp/vehicles

# Ozoneにコピーされたファイルを確認します

$ ozone fs -ls ofs://ozone/vol1/bucket1/cp/vehicles

  • ozone fs -cpコマンドを使うと、ファイルは一旦クライアント側に全てダウンロードされてから再びクラスタへアップロードすることになり、非常に非効率です。
    hadoop distcpコマンドを使うと、HDFS上に分散して配置されているデータブロックを、そのまま並列転送するので効率的です。
    hadoop distcpはHDFS〜Ozone間のデータコピーにも対応しています。

$ ozone fs -mkdir -p ofs://ozone/vol1/bucket1/distcp/vehicles

# distcpを実行します。-m 2で、2つのタスクで並列コピーします

$ hadoop distcp -m 2 -skipcrccheck hdfs:///tmp/vehicles.csv ofs://ozone/vol1/bucket1/distcp/vehicles

# クライアント〜サーバ間の相互認証を使用する場合

# $ hadoop distcp  -Ddfs.data.transfer.protection=authentication -m 2 -skipcrccheck hdfs://cdp.54.170.129.255.nip.io:8020/user/admin/*  ofs://ozone/vol1/bucket1/distcp/vehicles

# Ozoneにコピーされたファイルを確認します

$ ozone fs -ls ofs://ozone/vol1/bucket1/distcp/vehicles

s3vボリューム

Ozoneには、初期状態で/s3vというボリュームが存在しており、S3GatewayによるS3アクセスの際に使用されます。

  • /s3vボリュームの情報を確認します。

$ ozone sh bucket link /vol1/obs-bucket /s3v/obs-bucket-link

s3vボリュームの注意事項

  • もしLEGACYバケットにS3アクセスする場合は、ozone.om.enable.filesystem.pathsをFalseに変更する必要があります。
  • 設定を変更するには、Ozoneサービルロールの設定画面で、”Ozone Service Advanced Configuration Snippet (Safety Valve)”を探し、以下のプロパティを設定します。
  • Name: ozone.om.enable.filesystem.paths
  • Value: False
  • Ozoneをリスタートします
  • S3Gateway経由でFSOバケットにアクセスすることも可能です。
    しかし、S3プロトコルとの互換性の問題で、ディレクトリ作成は失敗します。

S3Gatewayの利用

  • S3GatewayのTLSが有効化されている場合、以下の手順で証明書をエクスポートします。

# TLSアクセスするためのCAのエイリアス名を確認

$ keytool -list -v -keystore /opt/cloudera/security/jks/truststore.jks

Enter keystore password:

Keystore type: PKCS12

Keystore provider: SUN

Your keystore contains 1 entry

Alias name: ca-cert.pem     # <===== エイリアス名

Creation date: Sep 5, 2025

(省略)

# CA証明書をエクスポート

$ keytool -export -alias ca-cert.pem –file s3g-ca.crt -keystore /opt/cloudera/security/jks/truststore.jks

Enter keystore password:

Certificate stored in file <s3g-ca.crt>

# CA証明書をPEMフォーマットへ変換

$ openssl x509 -inform DER -outform PEM -in admin-s3g-ca.crt -out /tmp/s3gca.pem

$ ls -l /tmp/s3gca.pem

-rw-r--r-- 1 admin admins 1651 Sep  4 07:59 /tmp/s3gca.pem
